1. 백엔드 생태계와 Java의 엔터프라이즈 점유율

Python vs Java:

Python은 풍부한 라이브러리를 바탕으로 데이터 사이언스와 AI 영역에서 압도적인 효율을 발휘한다.

반면 Java는 강력한 타입 체크, 안정성, 그리고 Spring Framework라는 거대한 생태계를 바탕으로 대규모 트래픽 처리가 필요한 엔터프라이즈급 백엔드(금융권, 공공기관 산내 시스템 등)의 *De facto standard(사실상 표준)*로 자리 잡고 있다.

2. 객체지향 프로그래밍 (OOP)의 본질

OOP (Object-Oriented Programming): 실세계(Real World)의 유형/무형 사물(Object)들을 추상화(Abstraction)하여 프로그램의 메모리 영역으로 모델링하는 패러다임.

Class와 Instance의 구분:

Class: 실세계의 객체를 프로그램으로 끌고 오기 위한 '설계도(Template)'. 명사적 특징인 상태(Variable/Property)와 동사적 특징인 행위(Method)로 구성된다.

Instance: 추상적인 클래스 설계도를 바탕으로, 실제 프로그램 실행 시(Runtime) 메모리에 할당되어 동작하는 실체. (통상적으로 Object와 혼용해서 쓰지만, 메모리에 로드된 상태를 강조할 때 Instance라 부른다.)

3. Java의 핵심 아키텍처: JVM과 OS 독립성

C/C++ 같은 언어가 컴파일 시 해당 운영체제(OS)가 이해할 수 있는 기계어로 번역되어 OS에 종속(Dependent)되는 반면, Java는 "Write Once, Run Anywhere"라는 철학을 가진다.

OS 인디펜던트 (Independent): Java 애플리케이션은 OS 위에서 직접 실행되지 않는다.

계층 구조 (JDK $\supset$ JRE $\supset$ JVM):

JVM (Java Virtual Machine): OS와 Java 프로그램 사이에서 중재자 역할을 하는 가상 머신. OS 종류(Windows, Mac, Linux)에 맞는 JVM만 설치되어 있다면, 컴파일된 Java 바이트코드(.class)는 어디서든 동일하게 실행된다.

JRE (Java Runtime Environment): JVM + 자바 코드를 실행하는 데 필요한 기본 라이브러리(API).

JDK (Java Development Kit): JRE + 개발 툴(컴파일러 javac, 디버거 등). 개발을 위해서는 반드시 JDK가 필요하다.

4. JDK 벤더사와 버전(LTS) 호환성 트레이드오프

자바는 Oracle이 인수한 이후 상용화 정책에 변화가 생기면서, 무료 오픈소스 기반의 OpenJDK (Eclipse Adoptium/Temurin 등)를 실무에서 널리 사용하고 있다.

버전 선택의 고민:

현재 실무(Legacy)에서 가장 안정적으로 많이 쓰이는 LTS 버전은 Java 11이다.

하지만 최근 모던 백엔드 아키텍처를 구성하는 Spring Boot 3.x 버전 이상은 Java 17 이상을 강제한다. 따라서 교육 과정 및 신규 프로젝트 세팅 시에는 Java 17 혹은 21 버전을 채택하는 것이 기술 부채를 줄이는 올바른 접근이다.
