1. React의 본질: Library와 제어권 (IoC)

React 공식 문서에서는 리액트를 "사용자 인터페이스를 빌딩하기 위한 자바스크립트 라이브러리"라고 정의한다.

💡 고민 포인트 (Library vs Framework): Vue나 Angular 같은 '프레임워크'는 애플리케이션의 뼈대와 흐름을 프레임워크가 주도(제어의 역전, IoC)하는 반면, React는 철저히 UI 렌더링에만 집중하는 '라이브러리'다. 즉, 라우팅이나 상태 관리 등은 개발자가 필요에 따라 직접 서드파티 라이브러리를 호출(제어권을 가짐)하여 조립하는 생태계를 가진다.

2. 컴포넌트(Component) 기반 아키텍처와 재사용성

과거 순수 HTML/CSS/JS 방식은 페이지 단위의 마크업 중심이었다면, React는 UI를 레고 블록과 같은 '독립적인 부품(Component)'으로 분리하여 조립하는 방식(Component-Based)을 채택한다.

아키텍처 관점의 재사용성:

에어비앤비(Airbnb)의 숙소 리스트를 예로 들면, 수십 개의 숙소 카드가 있지만 이를 그리는 '카드 컴포넌트(틀)'는 단 하나다.

이 컴포넌트가 '의존성이 없는 독립적인 모듈'로 설계되었을 때만 다른 페이지나 심지어 다른 프로젝트에서도 완벽한 재사용이 가능하다. (강한 결합도를 피하는 설계 원칙과 동일)

3. 컴포넌트 파이프라인: Props ➡️ Component ➡️ Element

컴포넌트는 함수(Function)다. 자바스크립트 함수가 입력을 받아 출력을 내듯, React 컴포넌트도 동일한 파이프라인을 거친다.

Input (Props): 부모로부터 전달받는 읽기 전용 속성(Property). 붕어빵의 앙금(팥, 슈크림)에 비유할 수 있다. 에어비앤비에서는 이미지 URL, 지역명, 거리가 Props가 된다.

Process (Component): Props를 가공하여 화면에 그릴 준비를 하는 붕어빵 틀. 최신 React 생태계에서는 클래스형 컴포넌트보다 '함수형 컴포넌트(Function Component)'를 사용하는 것이 사실상 표준(De facto standard)이다.

Output (Element): 컴포넌트를 통해 최종적으로 메모리에 생성되어 브라우저 화면(DOM)에 렌더링될 준비를 마친 최소 단위의 블록. (완성된 붕어빵)

4. 렌더링(Rendering)이란?

생성된 Element들이 브라우저 파서(Parser)에 의해 해석되어 실제 브라우저 화면(DOM Tree)에 시각적으로 그려지는 과정을 의미한다.

React는 SPA(Single Page Application) 기반이므로, 페이지 전체를 새로고침(Reload)하지 않고 데이터(Props, State)가 변경된 컴포넌트만 가상 DOM(Virtual DOM)을 통해 추적하여 효율적으로 리렌더링(Re-rendering)하는 강력한 아키텍처를 가진다.

💡 오늘의 회고

오늘 하루 중 얼마나 몰입했는지:

단순히 "리액트는 컴포넌트 단위다"라고 외우던 것에서 벗어나, Props가 주입되어 Element가 튀어나오고 그것이 브라우저 DOM에 렌더링되는 일련의 파이프라인을 함수의 입출력 개념과 맵핑하며 깊게 몰입할 수 있었다.
