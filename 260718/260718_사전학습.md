1. 패키지(Package)를 통한 네임스페이스와 모듈화

단순한 자바 문법을 넘어 엔터프라이즈 환경에서는 수많은 클래스 파일이 생성된다. 이를 하나의 디렉토리에 몰아넣는 것은 유지보수성을 극도로 떨어뜨리며 네이밍 충돌을 유발한다.

따라서 관련된 클래스들을 논리적인 패키지(Package)로 묶어 관리하며, 다른 패키지의 클래스를 사용할 때는 import 키워드를 통해 의존 관계를 명시한다. 실무에서는 보통 특정 도메인(e.g., user, board)을 최상위 패키지로 두고, 그 하위에 아키텍처 계층별 폴더를 두는 방식을 취한다.

2. Spring MVC와 3계층 아키텍처 (Layered Architecture)

웹 애플리케이션의 복잡도를 낮추기 위해 시스템을 역할에 따라 3가지 논리적 레이어로 분리(Separation of Concerns)한다.

Controller (제어 계층):

농구의 포인트 가드나 축구의 미드필더처럼 전체적인 흐름을 조율한다.

클라이언트의 HTTP 요청을 가장 먼저 받아 분석하고, 적절한 Service 로직을 호출한 뒤, 그 결과를 다시 클라이언트에게 반환(주로 JSON 형태)하는 역할만 전담한다.

Service (비즈니스 로직 계층):

애플리케이션의 핵심 비즈니스 로직(업무 처리)이 수행되는 영역. 트랜잭션의 단위가 되기도 한다.

Repository / DAO (데이터 영속성 계층):

데이터베이스(DBMS)와의 통신을 전담하는 계층.

JPA(ORM) 기반일 때는 주로 Repository라는 네이밍을, MyBatis(SQL Mapper) 기반일 때는 DAO(Data Access Object)라는 네이밍을 사용한다.

3. 레이어 간 데이터 전달: DTO와 Entity

각 레이어 간 데이터를 주고받을 때는 도메인 객체를 사용한다.

Entity: DB 테이블과 1:1로 매핑되는 객체로, JPA를 사용할 때 영속성 컨텍스트에 의해 관리된다. (DB 관점)

DTO (Data Transfer Object): 레이어 간(주로 Controller ↔ Service, 혹은 클라이언트 ↔ Controller) 데이터를 주고받기 위한 순수 데이터 컨테이너. 로직을 갖지 않으며 요청(RequestDTO)과 응답(ResponseDTO) 스펙에 맞게 분리하여 사용한다.

4. 의존성과 고응집 저결합 (High Cohesion, Loose Coupling)

Controller는 자신이 처리할 수 없는 비즈니스 로직을 Service에게 위임해야 하므로, 내부 멤버 변수로 Service 인스턴스를 가져야 한다. (e.g., TestService service = new TestService();)

💡 CS 설계 철학:

응집도(Cohesion)는 높게: 각 클래스는 자신의 목적에 맞는 하나의 기능에만 집중해야 한다. (단일 책임 원칙)

결합도(Coupling)는 낮게: 클래스 간의 의존성이 너무 강하면 하나의 변경이 연쇄적인 수정을 야기한다. 따라서 인터페이스나 프레임워크의 DI(Dependency Injection)를 통해 결합도를 낮추는 유연한 설계가 요구된다.

💡 오늘의 회고

학부 시절 소프트웨어 공학 시간에 추상적으로만 배웠던 '고응집 저결합'이나 'Layered Architecture' 패턴이 Spring 웹 백엔드 프로젝트의 실제 디렉토리 구조와 클래스 네이밍(Controller, Service, Repository)으로 어떻게 직결되는지 뼈저리게 깨달으며 엄청난 몰입감을 느꼈다.

오늘 영상에서 Controller가 Service를 의존하기 위해 new 키워드로 직접 객체를 생성하는 코드를 작성했지만, 실제 Spring Framework의 핵심 기술인 IoC(제어의 역전) 컨테이너와 DI(의존성 주입)를 활용하면 이 new 키워드를 어떻게(@Autowired나 생성자 주입 등으로) 없애고 결합도를 낮출 수 있는지 추가로 선행 학습을 해보아야겠다.
