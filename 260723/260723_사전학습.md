📝 공부한 내용 본인의 언어로 정리하기

1. SPA 라우팅의 본질: 새로고침 없는 화면 전환 (CSR)

Hard Routing vs Soft Routing: 전통적인 웹(window.location.href 등)은 URL이 변경되면 서버에 완전히 새로운 .html 문서를 요청하는 Hard Routing(새로고침)이 발생한다. 이 경우 메모리에 올려둔 상태(State)가 모두 날아간다.

CSR (Client-Side Routing): React는 SPA이므로, HTML5의 History API를 활용하여 브라우저의 URL 주소만 변경하고 서버에 요청을 보내지 않는다. 대신 변경된 URL 경로(Path)를 가로채어, 해당 경로에 매핑된 컴포넌트(Virtual DOM)만 브라우저 화면에 새로 마운트(Mount)한다. 이를 구현하는 표준 라이브러리가 react-router-dom이다.

2. 프로그래밍 방식의 라우팅과 상태 전달 (useNavigate & useLocation)

컴포넌트 내에서 특정 이벤트(예: 로그인 성공) 발생 시 강제로 화면을 전환하기 위해 useNavigate 훅(Hook)을 사용한다.

데이터의 은닉과 전달 (Routing State):

로그인한 유저의 ID(userName)를 다음 컴포넌트(Welcome.jsx)로 넘겨야 할 때, URL에 쿼리 스트링(Query String)으로 노출하는 것은 보안 및 UX 상 좋지 않다.

navigate('/welcome', { state: { userName: userId } })처럼 두 번째 인자로 state 객체를 넘기면, 브라우저 메모리(History API State)를 통해 데이터를 은닉하여 안전하게 전달할 수 있다.

전환된 컴포넌트에서는 useLocation().state를 통해 이 데이터를 꺼내어 렌더링에 활용할 수 있다. (Redux 같은 전역 상태 관리 도구 없이도 뎁스가 얕은 컴포넌트 간 데이터 전달이 가능한 강력한 방법이다.)

3. 컴포넌트 스타일링 아키텍처 (Inline vs External)

이전 Login.jsx에서는 관심사의 분리(SoC)를 위해 별도의 .css 파일을 Import하여 className으로 매핑했다.

반면 Welcome.jsx 실습에서는 컴포넌트 내부에 JS 객체(Object) 형태로 스타일을 정의하고 style={styles.container} 방식으로 바인딩하는 Inline Styling을 적용했다.

💡 아키텍처 고민 (Trade-off): 외부 CSS 파일은 브라우저 캐싱과 재사용성에 유리하지만 컴포넌트 간 클래스명 충돌 위험이 있다. 반면, Inline 객체 스타일링이나 Styled-components 같은 CSS-in-JS 방식은 스타일이 해당 컴포넌트에 완전히 종속(Scoped)되므로 모듈화와 동적 스타일링(State에 따른 색상 변경 등)에 압도적으로 유리하다는 점을 상기했다.
