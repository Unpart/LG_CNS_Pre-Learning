1. 모던 프론트엔드와 Node.js 런타임

왜 프론트엔드에 Node.js가 필요한가?: 자바스크립트는 원래 브라우저 내부에서만 동작하던 언어였으나, Node.js라는 '런타임(Runtime)'이 등장하면서 브라우저 밖(로컬 OS)에서도 JS를 실행할 수 있게 되었다. 모던 프론트엔드 환경(React, Vue 등)은 로컬에서 코드를 번들링(빌드)하고 개발용 웹 서버를 띄우기 위해 이 Node.js 환경을 필수로 요구한다.

npm vs npx (패키지 관리 아키텍처):

npm (Node Package Manager): 라이브러리를 로컬 환경에 영구적으로 다운로드하고 관리하는 툴.

npx (Node Package eXecute): 패키지를 로컬에 설치하지 않고, 원격(Registry)에서 최신 버전의 바이너리를 임시로 다운받아 실행(Execute)만 한 뒤 지워버리는 도구. npx create-react-app을 사용하면 항상 최신 버전의 React 보일러플레이트를 생성할 수 있어 스토리지 낭비와 버전 충돌을 방지한다.

2. React SPA 아키텍처 및 렌더링 파이프라인 분석

CRA로 생성된 프로젝트의 디렉토리 구조를 분석하며, SPA(Single Page Application)가 화면을 렌더링하는 내부 파이프라인을 뜯어보았다.

Phase 1: 껍데기 준비 (public/index.html)

브라우저가 최초이자 유일하게 로드하는 정적 HTML 파일.

내부를 보면 바디 영역에 오직 <div id="root"></div>라는 비어있는 식별자 태그 하나만 존재한다.

Phase 2: 브릿지 연결 및 렌더링 (src/index.js)

바닐라 JS의 document.getElementById('root')를 사용해 HTML의 빈 껍데기 노드를 캡처한다.

이 노드를 React의 가상 DOM(Virtual DOM) 루트로 생성하고, 그 안에 최상위 컴포넌트인 <App />을 주입(Injection/Render)하여 실제 브라우저 DOM 트리에 마운트시킨다.

Phase 3: UI 선언 (src/App.js)

화면에 그려질 실질적인 UI와 로직을 담고 있는 함수형 컴포넌트(Function Component).

Javascript 내부에서 HTML 문법을 사용할 수 있게 해주는 JSX 확장자를 반환(Return)하며, 이 반환된 엘리먼트들이 최종적으로 화면에 렌더링된다.

3. 트러블슈팅 및 터미널 환경 최적화

CLI 환경 통일: 윈도우 기본 CMD나 PowerShell은 명령어 처리에 호환성 이슈가 발생할 수 있으므로, 리눅스 명령어 체계를 지원하는 Git Bash를 VS Code의 기본 통합 터미널로 설정하여 안정성을 높였다.

npm start 명령어 인식 오류: Node.js 설치 직후 VS Code 통합 터미널에서 npm 명령어를 찾을 수 없다는 에러(command not found)가 발생할 수 있다. 이는 OS에 새롭게 등록된 환경 변수(Path)를 VS Code 터미널 프로세스가 아직 갱신하지 못해 발생하는 문제로, VS Code를 완전히 재시작하여 해결했다.

💡 오늘의 회고

어제 이론으로만 배웠던 "리액트는 SPA 기반이며 컴포넌트를 레고 블록처럼 끼워 넣는다"는 추상적인 개념이, 오늘 index.html의 빈 div 태그와 index.js의 render 함수 코드를 눈으로 직접 확인하는 순간 완벽하게 퍼즐이 맞춰지며 엄청난 쾌감과 몰입을 느꼈다.
