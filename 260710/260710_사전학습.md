파일 상태 변경(Untracked, Modified, Deleted)과 Staging

History Rewrite: git reset (--hard)

Forward-rolling Undo: git revert

Context Isolation: Branch (생성, 이동, 삭제)

📝 공부한 내용 본인의 언어로 정리하기

1. 파일의 상태 변화와 커밋 옵션의 한계

파일을 단순히 수정(Modify)하거나 삭제(Delete)하는 것 외에도, 새로운 파일을 생성하면 Git은 이를 Untracked(추적되지 않음) 상태로 인식한다.

💡 git commit -am의 함정: -a(all) 옵션을 사용하면 add와 commit을 동시에 수행할 수 있어 편리하지만, Untracked 상태인 '새로 생성된 파일'은 스테이징 영역에 포함시키지 못한다. - 따라서 신규 파일이 추가된 경우에는 반드시 git add .를 통해 모든 변경/추가 사항을 명시적으로 스테이징한 후 커밋해야 오류를 방지할 수 있다.

2. 과거로의 회귀: Reset vs Revert (핵심 트러블슈팅)

코드에 치명적인 버그가 발생해 이전 정상 상태(Commit)로 되돌아가야 할 때, Git은 두 가지 아키텍처적 접근을 제공한다.

git reset --hard [해시값] (히스토리 삭제 방식):

타임머신을 타고 과거로 돌아간 뒤, 그 이후의 미래(커밋 이력)를 완전히 삭제(Rewrite)해버린다. Working Directory의 코드도 과거 상태로 강제 복원된다.

주의점: 나 혼자 쓰는 로컬 브랜치에서는 유용하지만, 이미 원격(GitHub)에 Push된 커밋을 Reset 하면 팀원들과의 히스토리가 어긋나 치명적인 충돌(Conflict)을 유발하므로 협업 시에는 지양해야 한다.

git revert [해시값] (히스토리 보존 방식):

과거의 특정 커밋 내용을 '역으로 수행(Undo)'하는 새로운 커밋을 생성하여 덧붙인다.

장점: 기존의 커밋 히스토리를 훼손하지 않고 유지하므로, 안전하게 코드를 롤백할 수 있어 실무 협업 환경에서 표준 권장되는 롤백 방식이다.

3. 독립된 작업 공간: 브랜치 (Branch)

Branch의 본질: 메인 프로젝트(Main)의 뼈대에 영향을 주지 않고, 새로운 기능 개발이나 버그 수정을 위해 파생시키는 격리된 작업 공간(Context Isolation)이다.

주요 명령어:

git branch [이름]: 브랜치 생성

git switch [이름]: 브랜치 이동 (Checkout을 대체하는 모던 명령어)

git switch -c [이름]: 생성과 동시에 해당 브랜치로 이동 (실무에서 가장 많이 씀)

git branch -D [이름]: 해당 브랜치 강제 삭제 (병합되지 않은 내용이 있어도 삭제)

💡 오늘의 회고

Reset과 Revert의 차이를 단순히 '되돌리기'로 뭉뚱그려 외우지 않고, "히스토리를 덮어쓰느냐(Reset) vs 정반대의 변경 사항을 담은 새 커밋을 추가하느냐(Revert)"라는 내부 아키텍처 관점에서 이해할 수 있어 매우 흥미로웠다. 특히 CLI 창에서 해시값을 다루다가 SourceTree의 GUI 그래프를 보니 파일의 추적 상태가 뇌리에 명확히 꽂히는 느낌이었다.

오늘 배운 브랜치(Branch) 생성 및 이동을 바탕으로, 내일은 서로 다른 브랜치에서 동일한 파일의 같은 라인을 수정했을 때 필연적으로 발생하는 '병합 충돌(Merge Conflict)' 상황을 의도적으로 재현하고, 이를 안전하게 해결(Resolve)하는 과정을 실습해 볼 계획이다.
