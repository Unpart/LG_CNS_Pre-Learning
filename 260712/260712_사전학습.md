1. Context Isolation과 병합 충돌(Merge Conflict)의 발생

충돌의 원리: 서로 다른 브랜치(격리된 차원)에서 동일한 파일의 같은 라인을 동시에 수정했을 때 발생한다. Git은 두 가지 변경 사항 중 어떤 것이 맞는 코드인지 알고리즘적으로 판단할 수 없기 때문에 병합(Merge/Rebase)을 일시 중단하고 개발자에게 수동 해결(Resolve)을 위임한다.

안전 장치 (Abort): 충돌이 발생했을 때 당장 해결하기 어렵거나 상황이 꼬였다고 판단되면, git merge --abort 또는 git rebase --abort 명령어를 통해 병합 시도 자체를 취소하고 이전 상태로 안전하게 돌아갈 수 있다.

VS Code를 활용한 직관적 해결: 충돌 시 파일 내부에는 <<<<<<< HEAD (현재 브랜치)와 >>>>>>> (대상 브랜치) 마커가 생성된다. VS Code의 GUI를 활용하면 '현재 변경 사항 수락(Current)', '수신 변경 사항 수락(Incoming)', '모두 수락(Both)' 중 하나를 직관적으로 선택하여 코드를 정리한 뒤 다시 git add와 git commit을 수행하여 병합을 마무리할 수 있다.

2. Git과 GitHub의 명확한 분리 (Local vs Cloud)

Git: 내 로컬 환경(PC)에서 작동하는 분산 버전 관리 엔진. (인터넷 없이도 커밋, 브랜치 생성 등 모든 기능 수행 가능)

GitHub: Git으로 관리되는 저장소를 클라우드에 호스팅하여 팀원 간의 협업(PR, Issue 트래킹), 배포 자동화(CI/CD)를 가능하게 하는 서비스 플랫폼.

3. 원격 저장소 동기화 및 비동기화(Out-of-sync) 해결 전략

원격 저장소(Remote)와 로컬 저장소(Local)를 연결하고 나면 push와 pull을 통해 상태를 동기화해야 한다.

🚨 핵심 트러블슈팅 (Push Rejected):

상황: A 개발자가 로컬에서 코드를 수정하고 커밋했다. 그런데 그사이 B 개발자가 먼저 원격 저장소(GitHub)에 새로운 코드를 push해둔 상태다.

문제: 이 상태에서 A가 무작정 git push를 시도하면, Git은 원격 저장소의 최신 이력이 덮어씌워져 날아가는 것을 방지하기 위해 Push를 강제로 거절(Reject)한다. (원격 저장소와 로컬 저장소의 히스토리가 어긋남)

해결 프로세스: 반드시 원격의 최신 상태를 먼저 로컬로 당겨와서(Fetch+Merge) 병합해야 한다. git pull --no-rebase 옵션을 사용하여 원격의 변경 사항을 로컬에 안전하게 머지한 후, 동기화가 완료된 상태에서 다시 git push를 수행해야 완벽한 협업 사이클이 완성된다.

💡 오늘의 회고

혼자 코딩할 때는 겪기 힘든 '병합 충돌(Conflict)'과 '원격 저장소 Push 거절' 상황을 의도적으로 발생시키고 해결하는 과정이 매우 흥미로웠다. 특히 Git이 얼마나 안전하게 협업자들의 코드가 유실되지 않도록 방어 로직(Reject)을 짜놓았는지 아키텍처적 관점에서 감탄하며 몰입할 수 있었다.

git pull 명령어 내부적으로는 git fetch와 git merge가 순차적으로 일어난다는 점을 배웠다. 내일은 단순히 pull을 치는 것을 넘어, fetch만 먼저 수행하여 원격 추적 브랜치(origin/main)의 상태를 눈으로 확인한 뒤 수동으로 merge 해보는 디테일한 과정을 직접 실습해 보아야겠다.
