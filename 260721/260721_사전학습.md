📝 공부한 내용 본인의 언어로 정리하기

1. 관심사의 분리(SoC) 패러다임 변화: 파일에서 컴포넌트로

과거 바닐라 웹 환경에서는 HTML(구조), CSS(스타일), JS(동작)를 각각의 파일 단위로 물리적으로 분리하는 것이 관례였다.

하지만 React 생태계에서는 '컴포넌트(Component)'라는 논리적 단위로 관심사를 분리한다. 따라서 src/components/pages/ 하위에 Login.jsx를 생성하고, 이 컴포넌트가 온전히 자신만의 스타일을 갖도록 Login.css를 import하여 하나의 모듈(부품)처럼 조립할 수 있는 아키텍처를 설계했다.

2. JSX (JavaScript eXtension)의 본질

JXX(JSX)는 자바스크립트 코드 내부에 HTML 마크업을 직접 작성할 수 있게 해주는 문법적 설탕(Syntactic Sugar)이다.

💡 아키텍처 관점: 브라우저는 JSX를 이해하지 못한다. 내부적으로 Babel 같은 트랜스파일러가 이 마크업을 파싱하여 React.createElement() 자바스크립트 객체 호출로 변환한다. 자바스크립트 예약어인 class 대신 className을 어트리뷰트로 사용해야 하는 이유가 바로 여기에 있다. JSX는 HTML이 아니라 JS 함수 실행의 반환값(엘리먼트)이기 때문이다.

3. 상태(State) 기반의 제어 컴포넌트 (Controlled Component)

React는 양방향 통신과 UI 동기화를 위해 State(상태)를 사용한다.

useState 훅(Hook)을 사용하여 사용자가 입력하는 id와 password 값을 컴포넌트의 상태 변수에 바인딩한다. 이렇게 하면 입력 폼(Input)은 React 시스템에 의해 제어되는 '제어 컴포넌트'가 되며, 데이터의 단방향 흐름(One-way Data Flow)을 유지하면서도 사용자의 입력에 즉각적으로 반응하는 UI를 그릴 수 있다.

4. 🚨 트러블슈팅: SPA의 생명줄, e.preventDefault()

문제 발생: npm start로 서버를 띄우고 <form> 내부의 <button type="submit">을 클릭했더니, 콘솔에 로그가 찍히자마자 사라지고 화면이 깜빡이며 새로고침(Reload)되는 문제가 발생했다.

원인 분석 (CS 관점): HTML <form> 태그의 기본 동작(Default Behavior)은 버튼 클릭 시 설정된 action 경로로 데이터를 HTTP 전송하고 화면을 새로고침하는 것이다. 하지만 React는 최초 한 번만 페이지를 로드하고 가상 돔(Virtual DOM)으로 변경된 부분만 그리는 SPA(Single Page Application) 아키텍처를 따른다. 브라우저가 화면을 강제로 새로고침해 버리면 메모리에 올라가 있던 React의 상태(State)가 모두 초기화되어 버린다.

해결 방안: 이벤트 핸들러 handleSubmit 내부 최상단에 e.preventDefault(); 코드를 주입하여 브라우저의 기본 submit 동작(이벤트 버블링 및 새로고침)을 강제로 차단했다. 이를 통해 SPA의 라이프사이클을 안전하게 유지하면서 로직만 비동기로 처리할 수 있게 되었다.

💡 오늘의 회고

단순히 "리액트를 쓴다"를 넘어, 왜 JSX를 쓰는지, 왜 className이라고 적어야 하는지, 그리고 e.preventDefault()가 SPA 아키텍처에서 얼마나 치명적으로 중요한 방어 코드인지를 CS 레벨에서 뜯어보며 엄청난 몰입감을 느꼈다.

강의 마지막에 강사님께서 프론트엔드의 State 데이터를 백엔드(Spring Boot)로 넘기기 위해 Axios나 Fetch API를 사용한다고 언급하셨다. 내일은 JSON 형식으로 데이터를 직렬화(Serialize)하여 REST API로 비동기 통신을 수행하는 HTTP 네트워크 로직을 선행 학습해 보아야겠다.
