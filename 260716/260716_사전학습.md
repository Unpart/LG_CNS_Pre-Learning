1. 클래스의 아키텍처: 상태(State)와 행위(Behavior)

객체지향(OOP)에서 클래스는 실세계의 사물을 모델링하기 위한 템플릿이다.

이 템플릿은 데이터를 담는 그릇인 '변수(상태, State)'와, 이 데이터를 가공하고 처리하는 '메서드(행위, Behavior)'로 구성된다. (요리에 비유하자면 변수는 재료와 그릇, 메서드는 요리하는 과정 자체다.)

2. 변수의 스코프(Scope)와 메모리 생명 주기

변수는 선언된 위치(블록 { })에 따라 가시성(Visibility)과 메모리에 머무는 생명 주기(Lifecycle)가 완전히 달라진다.

전역 변수 (Global/Member Variable/Field):

클래스 블록 바로 아래에 선언되는 변수.

클래스 내의 모든 메서드에서 공유하고 접근할 수 있으며, 객체(Instance)가 Heap 메모리에 생성될 때 함께 할당되어 객체의 상태를 나타낸다. (강사님의 비유: 중앙 무대에서 활동하며 지역에 영향을 미치는 국회의원)

지역 변수 (Local Variable):

특정 메서드 블록 내부에서 선언되는 변수.

해당 메서드가 Call Stack에서 실행될 때 생성되었다가, 메서드 종료(블록을 벗어남)와 함께 메모리에서 소멸된다. 외부 메서드에서는 절대 접근할 수 없다. (강사님의 비유: 자기 관할 구역에서만 권한을 가지는 시/구의원)

3. 메서드 시그니처 (Method Signature)와 4가지 아키텍처

메서드는 접근지정자 반환타입 메서드명(매개변수)의 형태(Signature)를 가진다. 인자(Parameter)의 유무와 반환값(Return)의 유무에 따라 논리적으로 4가지 매트릭스로 분류할 수 있다.

[ O ] 인자 받음 / [ O ] 값 반환 (가장 이상적, 보편적 형태)

[ O ] 인자 받음 / [ X ] 값 반환 없음 (void)

[ X ] 인자 없음 / [ O ] 값 반환

[ X ] 인자 없음 / [ X ] 값 반환 없음 (void)

💡 아키텍처적 고민: 메서드의 본질은 입력(Input)을 받아 로직을 처리하고 출력(Output)을 내어주는 '함수'의 역할이다. 따라서 외부로부터 데이터를 주입(Injection/Parameter)받아 결과를 반환(Return)하는 1번 형태가 데이터의 무결성과 모듈화 측면에서 가장 활용도가 높다. 반환값이 없을 때는 void 키워드를 명시한다.

4. OOP의 본질: 우리는 왜 클래스를 만드는가?

추상화(Abstraction): 실세계에 존재하는 복잡한 사물(Object)을 프로그램 메모리 영역으로 그대로 끌고 올 수는 없다. 따라서 사물의 핵심적인 명사적 특징(변수)과 동사적 특징(메서드)만 뽑아내어 설계도(Template/Blueprint)를 작성하는데, 이것이 바로 클래스(Class)다.

인스턴스화 (Instantiation): 이렇게 만들어진 추상적인 클래스 설계도를 바탕으로, 실제 프로그램 실행 시(Runtime) 메모리(Heap)에 실체화된 데이터 구조를 찍어내는 것을 인스턴스(Instance) 생성이라고 한다.

💡 오늘의 회고

변수의 종류를 단순히 외우는 것이 아니라, "중괄호 { } 블록을 벗어나면 메모리에서 소멸된다"는 스코프(Scope)의 개념을 런타임 메모리(Call Stack) 관점에서 이해하며 몰입할 수 있었다.

내일은 오늘 배운 이 추상적인 '클래스(설계도)'를 바탕으로, 실제 메모리 상에 '인스턴스(실체)'를 어떻게 생성하고(아마도 new 키워드를 사용할 것 같다) 주소값을 참조 타입(Reference Type) 변수에 어떻게 할당하는지 직접 코드로 구현하며 CS 지식을 실증해 보아야겠다.
