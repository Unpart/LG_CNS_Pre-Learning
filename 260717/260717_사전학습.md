1. 클래스(Class)와 인스턴스(Instance)의 명확한 분리

개념의 혼용 경계: 흔히 클래스와 객체, 인스턴스를 혼용하여 부르지만, 아키텍처 관점에서는 철저히 격리하여 구분해야 한다.

클래스(Class): 상태(State, 필드/변수)와 행위(Behavior, 메서드)의 명세를 담고 있는 정적인 설계도(Template/Blueprint)이다.

인스턴스(Instance): 클래스라는 설계도를 바탕으로 런타임(Runtime)에 동적 메모리 영역인 Heap에 실체화되어 고유한 상태값을 갖는 데이터 구조이다.

소유의 주체: 클래스 내부에 정의된 멤버 변수와 메서드는 클래스 자체가 소유하는 것이 아니라, 해당 클래스로부터 찍혀 나온 고유한 인스턴스의 소유가 된다. 따라서 인스턴스화 과정을 거치지 않으면 외부에서 해당 멤버들에 접근할 수 없다.

2. 사용자 정의 타입(User-Defined Type)과 주소 참조

참조 타입으로서의 클래스: 자바에서 제공하는 기본 8가지 Primitive Type을 제외하면 모두 참조 타입(Reference Type)이다. 우리가 정의한 Teacher 클래스 역시 새로운 데이터 타입을 직접 정의하는 '사용자 정의 타입'으로 동작한다.

메모리 참조 아키텍처 (Stack & Heap):

Teacher ta = new Teacher();


Teacher ta: Stack 영역에 Teacher 타입을 가리킬 수 있는 참조 변수(포인터) ta를 할당한다.

new Teacher(): JVM이 Heap 메모리 영역에 Teacher 인스턴스에 필요한 만큼의 물리 공간을 할당하고 생성자를 통해 인스턴스 객체를 생성한다.

= (할당 연산): Heap 영역에 생성된 인스턴스의 시작 주소값(Memory Address)을 Stack 영역의 참조 변수 ta에 저장한다.

3. 멤버 접근 연산자(.)와 메서드 파이프라인

도트(.) 연산자: Stack에 있는 참조 변수가 가진 주소값을 바탕으로, Heap에 할당된 실제 인스턴스의 메모리 주소를 찾아가 멤버(변수/메서드)를 가리키는 주소 참조 연산자이다.

메서드 호출과 반환 흐름:

String result = ta.teaching("Java");


매개변수 "Java"가 Stack 프레임을 통해 Heap 영역 내 인스턴스의 teaching 메서드 컨텍스트로 전달(Injection)된다.

메서드 내부에서 연산 및 문자열 결합(+)이 처리된 후, return 키워드를 만나 제어권과 최종 반환값이 호출부(TeacherApp)로 넘어가며 참조 변수 result에 대입된다.

💡 오늘의 회고

오늘 하루 중 얼마나 몰입했는지:

단순히 객체를 생성한다는 단순 코딩을 넘어, Stack 영역의 참조 변수와 Heap 영역의 인스턴스가 주소값(Address)으로 어떻게 연결되는지 자바의 메모리 레이아웃 구조와 엮어 분석하며 깊게 몰입했다. C의 포인터가 자바의 참조 타입에서 어떻게 JVM에 의해 추상화되고 숨겨졌는지 구조적 연관성을 정립할 수 있었던 유익한 시간이었다.

더 나은 미래를 위해 내일 어떤 공부를 진행할 것인지:

오늘 생성한 인스턴스는 한 개였지만, 여러 개의 참조 변수가 동일한 하나의 인스턴스를 가리킬 때 발생하는 '얕은 복사(Call by Reference/Value)'의 메모리 동작 방식을 자바 코드로 테스트하며 JVM의 값 전달 메커니즘을 보다 확실하게 정복해 보아야겠다.
