1. Java의 변수 타입과 메모리 관점의 이해

데이터를 담는 그릇인 '변수(Variable)'는 라벨지(Type)에 따라 메모리 할당 방식이 근본적으로 달라진다.

Primitive Type (기본 타입):

int, double, boolean, char 등.

변수가 선언된 메모리 공간(주로 Call Stack)에 '실제 데이터(Raw Value)'를 직접 저장한다.

💡 고민 포인트 (Memory Efficiency): 정수형을 굳이 byte, short, int, long으로 나눈 이유는 메모리 풋프린트(Memory Footprint)를 최적화하기 위함이다. 하드웨어 자원이 제한적이던 과거와 달리 모던 프로그래밍에서는 보편적으로 4바이트인 int를 기본으로 사용하지만, 대용량 트래픽 처리 시 메모리 효율을 위해 데이터의 범위에 맞는 타입을 선정하는 엔지니어링 역량이 여전히 요구된다.

Reference Type (참조 타입):

기본 타입을 제외한 모든 타입 (클래스, 배열, 인터페이스 등). String이 대표적이다. (Java에서 String은 첫 글자가 대문자인 '클래스'이다.)

실제 데이터는 동적 메모리 영역인 Heap에 생성되고, 변수(Stack 영역)에는 그 객체가 위치한 '주소값(Memory Address/Pointer)'이 저장된다.

2. 식별자 명명 규칙 (Naming Convention)과 가독성

컴파일러가 강제하진 않지만, 개발자 간의 원활한 협업을 위해 엄격히 지켜야 하는 De facto standard(사실상 표준)이다.

Class: PascalCase (VariableApp) - 첫 글자 대문자.

Variable / Method: camelCase (playGame) - 첫 글자 소문자, 이어지는 단어의 첫 글자는 대문자.

Boolean Prefix: 논리형(boolean) 변수는 주로 is, has, can 등의 접두사를 붙여 의도를 명확히 한다. (e.g., isMarried)

3. 변수의 스코프(Scope)와 생명 주기

전역 변수 (Global Variable / Field): 클래스 블록({}) 바로 아래에 선언된 변수. 클래스가 인스턴스화될 때 Heap 메모리에 함께 할당되며, 객체의 상태(State)를 나타내는 멤버 변수 역할을 한다.

지역 변수 (Local Variable): 메서드 내부 블록에서 선언된 변수. 메서드 호출 시 Call Stack에 생성되었다가 메서드 종료 시 함께 소멸된다.

4. 핵심 아키텍처: main 메서드가 없는 클래스의 의미

오늘 실습한 VariableApp 클래스는 실행의 진입점(Entry Point)인 public static void main(String[] args) 메서드를 가지고 있지 않았다.

실행 불가능성: main 스레드를 띄울 수 없으므로 java 명령어로 단독 실행(Runtime)할 수 없다. 컴파일(javac)만 가능하다.

💡 아키텍처적 당위성 (왜 만드는가?): 객체지향 프로그래밍(OOP)에서 실세계의 사물을 프로그램으로 모델링할 때, 모든 클래스가 애플리케이션의 실행 주체가 되는 것은 아니다. main이 없는 클래스는 그 자체로 데이터를 담고 행위를 정의하는 '설계도(Template/Blueprint)'의 역할을 수행하며, 추후 다른 실행 클래스에서 메모리에 로드(인스턴스화)하여 사용하기 위해 존재한다.

💡 오늘의 회고

C언어에서 포인터(Pointer)라는 개념으로 어렵게 배웠던 메모리 주소 참조가, Java에서는 'Reference Type(참조 타입)'이라는 이름으로 JVM이 안전하게 추상화하여 관리해 준다는 사실을 다시금 연결해 보며 매우 몰입할 수 있었다.

오늘 만든 '실행할 수 없는 템플릿 클래스(설계도)'를 바탕으로, 내일은 이 설계도를 이용해 메모리에 실제 객체(Instance)를 생성(new 키워드)하고 Heap 메모리에 어떻게 데이터가 적재되는지 그 과정을 직접 코드로 구현하고 테스트해 보아야겠다.
