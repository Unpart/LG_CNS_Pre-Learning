1. 프론트엔드 개발 환경 구축과 로컬 웹 서버의 필요성

IDE(통합 개발 환경)의 선택:

자바(Java) 엔터프라이즈 환경에서는 Eclipse나 IntelliJ가 주력으로 쓰이지만, 프론트엔드(HTML/CSS/JS, React, Vue 등) 생태계에서는 다양한 확장성(Extension)을 제공하는 VS Code가 사실상 표준(De facto standard)이다.

웹 서버 구동의 당위성 (💡매우 중요):

개발한 index.html 파일을 더블 클릭해서 여는 것은 file:// 프로토콜을 사용하여 브라우저가 로컬 파일을 단순히 읽어들이는 것이다.

하지만 실제 웹 서비스는 클라이언트와 서버가 HTTP 프로토콜을 통해 통신하므로, 로컬 환경에서도 이와 동일한 네트워크 환경(http://127.0.0.1:5500)을 에뮬레이션해야만 한다. 이를 위해 VS Code의 Live Server 확장 팩을 설치하여 정적 리소스를 호스팅하는 로컬 웹 서버를 구동했다.

2. 관심사의 분리(SoC)와 모듈화

HTML 뼈대(Structure), CSS 표현(Presentation), JS 동작(Behavior)을 하나의 파일에 몰아넣을 수도 있지만, 이는 유지보수성과 재사용성을 극도로 떨어뜨린다.

따라서 디렉토리 구조부터 html, css/, script/ 등으로 분리하고, <link rel="stylesheet" href="...">와 <script src="..."></script> 태그를 이용해 외부 파일을 모듈처럼 가져와 연결하는 아키텍처를 구성했다.

3. DOM 트리 구조와 식별자(Selector)의 목적

DOM 계층 구조와 가독성: HTML 태그들의 중첩(Nesting)은 곧 브라우저가 파싱할 DOM 트리(Tree)의 계층 구조를 의미한다. 코드 작성 시 철저한 들여쓰기(Indentation)를 지키는 것은 단순히 보기 좋게 하는 것을 넘어, DOM의 계층 관계를 명확히 파악하기 위한 엔지니어의 기본 소양이다.

CSS 선택자와 JS 식별자의 차이점:

Class (.) 선택자: 여러 요소에 공통된 스타일(CSS)을 재사용하기 위해 주로 사용한다. (e.g., .login-container)

ID (#) 선택자: 특정 요소에 유일한(Unique) 식별자를 부여하여, 자바스크립트가 해당 DOM 객체에 정확히 접근(document.getElementById())하고 이벤트를 바인딩하기 위해 주로 사용한다.

4. 이벤트 주도 프로그래밍 (Event-Driven) 맛보기

폼(Form) 요소에서 발생하는 onsubmit이나 버튼의 onclick 같은 사용자 이벤트를 감지하여 자바스크립트의 핸들러 함수가 실행되도록 연결했다.

JS 로직 내에서 DOM 객체의 value를 읽어와 조건문(if-else)으로 단순 로그인 분기 처리를 수행하고, window.location.href 객체를 조작하여 다른 페이지로 강제 라우팅(Redirect) 시키는 기초적인 동작 흐름을 구현했다.

💡 오늘의 회고

1~2일차의 이론에서 벗어나 VS Code를 세팅하고 실제 코드가 브라우저에 렌더링되는 과정을 눈으로 확인하니 몰입도가 확 올라갔다. 특히 단순히 HTML을 여는 것과 Live Server를 띄워 HTTP 포트를 통해 접근하는 것의 아키텍처적 차이를 분명히 인지하며 코딩할 수 있어서 유익했다.

오늘 실습한 바닐라 JS(Vanilla JS)를 통한 직접적인 DOM 조작 로직이 프로젝트 규모가 커졌을 때 어떤 성능적 한계(리플로우, 리페인트 등)를 가져오는지 구글링해 보고, 추후 배우게 될 React의 Virtual DOM 개념과 연결 지어 이해해 볼 계획이다.
