1. JSX의 스타일링과 선언적 UI의 이점

JSX 아키텍처 리마인드: React 컴포넌트 내부에서 HTML처럼 보이는 코드는 사실 Babel에 의해 파싱되는 자바스크립트 객체다. 따라서 JS 예약어인 class 대신 className 어트리뷰트를 사용하여 CSS를 매핑해야 한다.

선언적 에러 핸들링: 바닐라 JS에서는 에러 발생 시 document.getElementById로 DOM을 찾아 직접 텍스트나 스타일을 주입해야 했다. 하지만 React에서는 useState로 에러 상태(errorMsg)만 변경해주면, React의 Virtual DOM이 알아서 변경된 상태를 감지하고 UI를 재조정(Re-rendering)한다. (데이터 상태가 UI를 결정하는 선언적 프로그래밍의 정수)

2. 백엔드 연동을 위한 비동기 통신 아키텍처 (Axios)

Axios의 도입: SPA 환경에서는 폼(Form)을 submit할 때 페이지를 새로고침하며 데이터를 전송하지 않는다. 대신, 백엔드 REST API와 JSON 포맷으로 데이터를 주고받기 위해 Axios라는 프로미스(Promise) 기반 HTTP 클라이언트 라이브러리를 사용한다.

async/await 동기화 제어: 네트워크 통신은 대표적인 비동기(Asynchronous) I/O 작업이다. 서버에 요청을 보낸 후 응답이 올 때까지 메인 스레드가 멈추지 않도록 해야 하며, 응답 데이터를 활용하기 위해 순서를 보장해야 한다. 따라서 이벤트 핸들러에 async를 선언하고, axios.post 요청 앞에 await를 걸어 프로미스가 Settled 될 때까지 코드 실행 흐름을 안전하게 제어했다.

인증(Auth) 토큰 관리: 백엔드로부터 응답(Response)받은 인증 토큰(JWT 등)을 브라우저의 localStorage.setItem()을 이용해 로컬 스토리지에 저장하는 로직의 뼈대를 구현했다. 이는 향후 권한이 필요한 API 요청 시 헤더(Header)에 토큰을 실어 보내기 위한 필수 과정이다.

3. SPA의 화면 전환: Client-Side Routing (React Router DOM)

전통적 라우팅 vs CSR: 과거의 웹은 링크를 클릭하면 서버에 새로운 .html 파일을 요청하여 화면 전체를 렌더링했다. 하지만 React(SPA)는 index.html 단 하나의 파일만 존재하므로, URL 변경을 감지하여 서버 요청 없이 브라우저 단에서 컴포넌트만 갈아 끼우는 클라이언트 사이드 라우팅(CSR)이 필요하다.

React Router DOM 구축: 이를 위해 react-router-dom 패키지를 설치하고, 최상위 컴포넌트인 App.js를 라우터 허브로 리팩토링했다.

<BrowserRouter>: HTML5의 History API를 사용하여 브라우저의 주소창(URL)을 제어.

<Routes> & <Route>: 현재 URL 경로(Path)에 매칭되는 특정 컴포넌트(예: / -> <Login />, /welcome -> <Welcome />)를 찾아 Virtual DOM에 주입(Mount)하는 역할을 수행한다.

💡 오늘의 회고

단순한 화면 그리기를 넘어, Axios를 통한 네트워크 I/O 비동기 제어와 React Router DOM을 활용한 SPA 라우팅 트리를 직접 구축하며 프론트엔드 아키텍처의 큰 그림이 완성되는 느낌에 엄청난 몰입감을 경험했다. 특히 async/await로 비동기 흐름을 제어하는 과정이 CS 지식과 맞물려 매우 흥미로웠다.

오늘은 라우팅 구조의 뼈대만 잡았으므로, 내일은 실제로 로그인에 성공했을 때 useNavigate 훅 등을 사용하여 /welcome 컴포넌트로 강제 라우팅(Redirect) 시키는 동적 화면 전환 로직을 구체적으로 실습해 보아야겠다.
