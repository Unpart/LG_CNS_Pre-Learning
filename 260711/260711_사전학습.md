1. 브랜치(Branch): 차원의 분리와 독립성 (Context Isolation)

실무 프로젝트는 절대 하나의 메인 라인(Main)에서만 개발되지 않는다. 신규 기능 개발, 버그 픽스, 테스트 환경 등을 위해 기존 코드의 베이스라인을 유지한 채 파생되는 독립적인 작업 공간이 필요한데, 이것이 바로 브랜치(Branch)다.

브랜치를 생성(git branch)하고 이동(git switch)하는 과정은 단순히 폴더를 복사하는 것이 아니라, Git 내부적으로 특정 커밋 해시(Hash)를 가리키는 '포인터(Pointer)'를 이동시키는 가벼운 작업이다.

2. 브랜치를 합치는 두 가지 패러다임: Merge vs Rebase

분기된 브랜치에서 작업이 끝난 후, 이를 다시 Main 브랜치로 통합하는 과정은 아키텍처 관점에서 두 가지 방식(Trade-off)이 존재한다.

방식 1: 3-Way Merge (git merge) - "히스토리의 완벽한 보존"

동작 원리: Main 브랜치에서 대상 브랜치(e.g., add-coach)를 병합하면, 두 브랜치의 변경 사항을 모두 합친 새로운 '병합 커밋(Merge Commit)'이 생성된다.

특징: 언제, 어떤 브랜치에서 분기되어 어떤 작업 후 병합되었는지 DAG(방향성 비순환 그래프) 히스토리가 있는 그대로 남는다. 안전하지만, 프로젝트 규모가 커지면 커밋 트리가 매우 복잡해질 수 있다.

방식 2: Rebase (git rebase) - "히스토리의 단일 선형(Linear) 구조화"

동작 원리: 병합할 브랜치(e.g., new-teams)의 베이스(시작점)를 Main 브랜치의 가장 최신 커밋으로 '재배치(Re-base)'한다. 기존 커밋들을 복사하여 Main의 끝단에 순서대로 새로 이어 붙이는(History Rewrite) 방식이다.

Fast-Forward의 필요성: Rebase를 완료한 직후, Main 브랜치의 포인터는 아직 과거에 머물러 있다. 따라서 Main으로 다시 스위치한 후, git merge [브랜치명]을 통해 Main의 포인터를 최신 커밋 위치로 당겨오는 Fast-Forward(빨리 감기) 작업이 필수적이다.

💡 주의점: Rebase는 커밋 트리를 하나의 일직선으로 깔끔하게 유지해주지만, 커밋 해시값이 완전히 새로 생성(재작성)되므로 원격 저장소(GitHub)에 이미 공유된 브랜치에는 절대 사용하면 안 된다는 실무적 원칙을 다시 한번 상기했다.

💡 오늘의 회고

Merge와 Rebase의 차이를 단순히 '합치는 명령어'로만 외웠을 때는 헷갈렸는데, 강사님이 설명해주신 '시간 여행의 시점(Base)'과 'Fast-Forward'의 개념을 적용해 CLI 터미널 그래프와 SourceTree GUI를 대조해가며 보니 내부 동작 원리가 완벽히 이해되었다.

브랜치를 합칠 때, 서로 다른 브랜치에서 동일한 파일의 같은 라인을 수정했다면 Git은 어떤 코드를 선택해야 할지 몰라 '병합 충돌(Merge Conflict)'을 발생시킨다. 내일은 이러한 충돌 상황을 의도적으로 유발하고, VS Code를 활용해 충돌(Conflict)을 안전하게 해결(Resolve)하는 실무 프로세스를 집중적으로 실습해 보아야겠다.
