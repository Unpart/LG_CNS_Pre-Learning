1. 버전 관리 시스템의 진화와 Git의 본질

DVCS (분산 버전 관리 시스템):

개발은 혼자 하는 것이 아니라 협업의 과정이며, 파일의 단순 백업(final_final_real.txt 방식)을 넘어 소스 코드의 변경 이력을 추적하고 충돌을 제어하는 시스템이 필수적이다.

Git은 중앙 서버에만 의존하지 않고 각 개발자의 로컬 환경에도 완전한 저장소(Repository)의 복제본을 가지는 분산형 아키텍처(DVCS)를 띄고 있어, 오프라인 환경에서도 버전 관리가 가능하다. (시간과 공간/차원의 제어)

2. Git(로컬 엔진)과 GitHub(원격 호스팅)의 명확한 분리

Git: 내 로컬 컴퓨터에 설치되어 소스 코드의 버전을 관리하는 '엔진'이자 오프라인 도구.

GitHub: Git으로 관리되는 로컬 저장소들을 클라우드 환경에 업로드(Hosting)하여, 다른 팀원들과 코드 리뷰(PR), 이슈 트래킹, CI/CD(GitHub Actions) 등 실질적인 '협업'을 가능하게 해주는 온라인 플랫폼 서비스.

3. Git Lifecycle과 논리적 상태 변화

파일의 변경 사항이 버전으로 기록되기까지는 세 가지 논리적 영역을 거친다.

Untracked / Modified (Working Directory): 코드가 생성/수정되었으나 Git이 아직 추적하지 않거나 변경만 감지한 상태.

Staging Area (git add):

커밋할 파일들을 모아두는 논리적 대기 공간(타임캡슐).

💡 왜 바로 커밋하지 않고 스테이징을 거치는가? 변경된 수많은 파일 중 논리적으로 연관된(하나의 기능 단위 등) 파일들만 선별하여 하나의 커밋으로 묶기 위함이다.

Repository (git commit):

스테이징된 변경 사항들에 해시(Hash) 식별자를 부여하고 완전한 버전(스냅샷)으로 로컬 데이터베이스에 영구 기록(타임캡슐을 땅에 묻음)하는 과정.

4. CLI vs GUI 트레이드오프 및 기초 설정

기초 설정 (git config): 커밋 이력에 남을 작성자 식별을 위해 user.name, user.email을 전역(--global)으로 설정하고, 기본 브랜치명을 main으로 변경(init.defaultBranch)하는 초기 셋업을 진행했다.

CLI (Command Line Interface): Git Bash 등을 이용해 명령어로 제어. 속도가 빠르고 파이프라인 자동화에 유리.

GUI (Graphical User Interface): SourceTree 등을 이용. 복잡한 브랜치(Branch) 병합(Merge) 내역이나 커밋 히스토리를 시각적 그래프(DAG)로 한눈에 파악하기 좋음.

💡 오늘의 회고

단순히 깃 명령어만 외우던 과거와 달리, 강사님의 타임캡슐 비유(만들기=add, 묻기=commit)와 전공 지식을 결합해 Working Directory - Staging Area - Repository로 이어지는 Git의 아키텍처를 완벽히 이해하며 높은 몰입을 경험했다.

오늘은 로컬 영역(init, add, commit)까지만 다루었지만, 내일은 생성된 로컬 리포지토리를 GitHub 원격 리포지토리(Remote Repository)와 연동하고 push, pull, clone하는 네트워크 기반의 협업 사이클을 직접 실습해 보며 익혀야겠다.
